# 系统更新中心与集群更新

“系统更新中心”是当前版本比较新的能力，它把“检查版本”“发起更新”“查看任务”“管理节点”放到了设置页高级区里。

如果你还没把“本地发版 -> 服务器可见新版本 -> 后台远程更新”这条主线串起来，先看 [从本地发版到服务器远程更新](/help?article=deployment-release-and-remote-update&audience=admin)。

## 适用对象

- 管理员
- 有多节点或集群更新需求的部署环境

## 页面结构

### 总览配置

- 当前版本
- 最新版本
- 最近检查
- 当前状态
- 版本说明预览
- 公告同步预览
- 独立更新预检
- 版本源配置
- 默认更新范围与策略
- 创建更新任务

### 任务执行

- 看活跃任务
- 看单机任务阶段日志
- 看 preflight / verification / rollback 摘要
- 看批次进度
- 看按 agent / node 聚合的批次详情
- 看历史任务
- 支持失败重试、创建回滚任务或继续跟踪

### 节点状态

- 看更新代理心跳
- 管理节点排空
- 恢复接流

## 当前支持的范围与策略

### 范围

- `app`
- `worker`
- `cluster`

### 策略

- `in_place`
- `rolling`
- `drain_and_cutover`

## 排空切换的意义

排空切换不是简单重启，而是先把任务从节点上迁走，再做更新，尽量降低影响范围。

当前版本已经支持：

- 节点排空
- 调度器避开排空节点
- 排空后恢复接流

但仍要注意：

- 账号运行态热迁移不是所有场景都已完全无感
- 如果账号凭据只剩一次性登录态，切换后仍可能需要重新登录

## 独立预检

当前版本支持先执行一次独立预检，再决定是否创建任务。预检会重点检查：

- 目标版本是否可解析
- 部署目录剩余空间
- 回退预留空间
- MySQL / Redis 健康
- 运行中账号是否存在重新登录风险
- 宿主机更新代理是否在线

如果预检有阻断项，任务不会进入待执行态，页面会直接展示阻断项明细。

## 公告同步与版本说明

“检查更新”后，系统更新中心会同时拉取版本说明预览，并给出“同步公告”建议。

- 版本说明优先使用 release cache
- 如果运行环境没有 `Update.log`，会回落到内置 release notes
- 更新任务可以选择“更新后同步公告”

这样管理员不需要再切到公告管理页手动确认一次。

## 回滚入口

当某次更新任务已经结束，并且存在可回退的版本信息时，任务卡片会出现“创建回滚任务”入口。

- 回滚同样以标准任务方式执行
- 回滚任务也会记录 phase、日志和核验结果
- 如果开启了“失败后提示回滚”，失败任务会在详情区给出回滚建议

## 宿主机更新代理

系统更新中心依赖宿主机上的更新代理脚本协同工作。

### 单次执行

```bash
/opt/qq-farm-current/update-agent.sh --once
```

### 常驻服务

```bash
/opt/qq-farm-current/install-update-agent-service.sh
```

## 远程更新前提

要让“系统更新中心”真的能更新服务器，至少要满足这几件事：

- 版本源里已经有新版本，不能只是你本地改完代码
- 宿主机部署目录里有最新的更新脚本
- `qq-farm-update-agent` 已安装并在线

如果你不确定脚本是不是新的，可以先在宿主机执行：

```bash
cd /opt/qq-farm-current
bash repair-deploy.sh --backup
bash install-update-agent-service.sh
systemctl status qq-farm-update-agent
```

## 正常更新顺序

单机环境最推荐按下面顺序走：

1. 先点“检查更新”，确认最新版本已经被识别出来。
2. 需要的话先点“同步公告”，把版本说明和公告补齐。
3. 先执行“独立预检”，不要跳过阻断项。
4. 预检通过后，再创建更新任务。
5. 跟踪阶段日志、verify 结果和最终状态。
6. 如果失败且存在回退候选，优先直接创建回滚任务。

## 推荐的 smoke 验收

在正式发版或大版本升级前，建议先跑一遍非破坏性的联动 smoke：

```bash
cd /opt/qq-farm-current
bash smoke-system-update-center.sh \
  --username admin \
  --password '你的管理员密码' \
  --deploy-dir /opt/qq-farm-current
```

它会检查更新中心、公告同步 dry-run、独立预检、最近任务详情和宿主机核验，并把结果写到 `reports/system-update-smoke/<timestamp>/SUMMARY.md`。如果传了 `--deploy-dir`，脚本会自动读取对应 `.env` 里的 `WEB_PORT` 来拼出默认 `base-url`。

如果宿主机没有安装 `node`，当前版本会自动借用正在运行的主程序容器里的 `node` 完成 JSON 解析；前提是 Docker 可用，且应用容器已经正常启动。

## 建议

- 单机环境可以把它当成“可视化更新入口”。
- 多节点环境要先明确每个代理托管了哪些节点，再创建任务。
- 真正执行 `drain_and_cutover` 前，先看预检结果，不要跳过阻塞信息。
